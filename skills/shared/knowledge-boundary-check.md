# Knowledge Boundary Check

> 来源：主要从 `skills/project-init.md` 与 `skills/learn-plan.lisp` 拆出。此文件用于生成整周 / 整天学习包前的固定审查。

## 触发时机

- 生成整周学习包前
- 生成整天学习包前

单独补一个 lesson 或单个练习文件时，不强制触发整套流程。

## 审查步骤

1. 先读取当前计划中的主题、任务和显式知识点。
2. 列出本次允许出现的语法点、API、内置函数、方法、模块。
3. 列出学习者已经学过、可以直接复用的前置知识。
4. 扫描 lesson 示例区、练习示例区、练习题和检查标准中的所有调用。
5. 判断每个调用是否属于“已学知识”或“本次明确要学知识”。
6. 凡是不属于两者的内容，都标记为超纲。
7. 补洞完成后，重跑一次知识边界检查。
8. 生成后再次核对，确保没有漏网之鱼。

## 超纲判断

- 不属于“已学知识”或“本次明确要学知识”的 API / 内置函数 / 方法 / 模块，都视为超纲。
- 如果练习依赖某个未声明知识点，也视为超纲。

## 补洞策略

- 如果超纲点只有 1~3 个且绕不开，先在 lesson 中增加“先补 5 分钟”小节。
- 如果超纲点已经影响主线理解，先补一个独立前置知识块，再进入正式内容。
- 补洞内容只讲最小必要用法，不要顺势扩成额外长课。

## 生成后复核

- lesson 的示例区、练习文件的示例区、练习区、检查标准里出现的 API，都必须能归到：
  - 已学过
  - 今天新学
  - 今天先补
- 如果无法归类，回退并修改学习包。
